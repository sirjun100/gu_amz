# 客户端上报日志约定

本文约定 **EasyClick / 自动化客户端** 在调用服务端 **`POST /api/v1/client/tasks/{task_id}/report`** 时，随表单字段 **`log_lines`**（JSON 字符串数组）传递的日志格式，以及服务端如何据此结案、写入环境与完成时间。

---

## 1. 接口与数据形态

- **路径**：`POST /api/v1/client/tasks/{task_id}/report`
- **表单字段**：
  - `device_id`（必填）：须与当前任务绑定设备一致。
  - `log_lines`（必填）：**JSON 序列化后的字符串数组**，数组元素为按时间顺序（或任意顺序，只要含义清晰）的**单行字符串**；每条一行内容，不要带未转义的换行拼接进单条（如需多行，应拆成多条或把换行写成 `\n` 放在该字符串内）。
  - `images`（可选）：截图等多文件上传。

服务端先将 `log_lines` 解析为 `list[str]`，追加写入 `task_logs` 表，再按本文规则解析**结案信息**（成功/失败、失败原因、环境、完成时间）。

---

## 2. 推荐：摘要行 `AMZ_REPORT`

在人类可读日志之后，请在 **`log_lines` 数组末尾**（建议**倒数第 1 条**，且**单独占一行**）增加一条**机器可读摘要**，格式为：

```text
AMZ_REPORT <JSON 对象>
```

- **前缀**（区分大小写）：固定为 `AMZ_REPORT `（`REPORT` 与左大括号之间有一个空格）。
- **`<JSON 对象>`**：紧跟前缀之后的 **单行** UTF-8 JSON **对象**（`{ ... }`），不要使用 pretty-print 多行 JSON。

### 2.1 JSON 字段约定

| 字段 | 必填 | 类型 | 说明 |
|------|------|------|------|
| `status` | **是** | 字符串 | `"success"` 或 `"failed"`（**大小写不敏感**）。 |
| `environment` | 否 | 字符串 | 执行环境名称：如机器名、实例 ID、代理出口描述等。服务端写入任务表 `task_environment`，并在管理端展示为「环境」。**最长 256 字符**（超长截断）。 |
| `finished_at` | 否 | 字符串 | 客户端认定的任务结束时间，**ISO 8601**。例如 `2026-03-31T12:00:00Z`、`2026-03-31T12:00:00+00:00`、`2026-03-31T08:16:12.500Z`。若格式合法则写入数据库 `finished_at`；**不合法或省略**时，服务端使用**收到本次上报时**的时间作为 `finished_at`。 |
| `error` | 否 | 字符串 | **失败**时的简短原因；若存在且非空，优先作为 `failure_detail`。 |

其它键可以放 JSON 内供排障或日后扩展；**当前服务端仅持久化**与上表对应语义相关的字段（以及失败时若无 `error` 则可能将整段 JSON 截断写入 `failure_detail`）。

### 2.2 解析顺序（服务端）

1. 自**数组尾部向前**扫描，找到**第一条**（时间上最后一条）**整行**以 `AMZ_REPORT ` 开头的元素；
2. 解析其后 JSON；若解析失败或根类型不是对象 → 任务**失败**，`failure_detail` 说明解析错误；
3. 若 `status` 为 `success`（忽略大小写）→ **成功**，`failure_detail` 置空；
4. 否则 → **失败**；优先使用 `error` 作为 `failure_detail`，否则使用 JSON 字符串的截断版本。

### 2.3 示例

**成功：**

```text
2026-03-31 08:14:00 打开 App
2026-03-31 08:14:05 搜索关键词完成
AMZ_REPORT {"status":"success","environment":"win-proxy-01","finished_at":"2026-03-31T08:15:00Z"}
```

**失败：**

```text
2026-03-31 08:14:00 点击超时
AMZ_REPORT {"status":"failed","environment":"mac-official","error":"click timeout","finished_at":"2026-03-31T08:16:12.500Z"}
```

**客户端拼装示例（Python）：**

```python
import json

lines = []
lines.append("人类可读日志……")
summary = {
    "status": "success",
    "environment": "my-device-env",
    "finished_at": "2026-03-31T08:00:00Z",
}
lines.append("AMZ_REPORT " + json.dumps(summary, ensure_ascii=False))
# POST log_lines=json.dumps(lines, ensure_ascii=False)
```

## 3. 兼容：旧版「最后一行 success」

若整段 `log_lines` 中 **不存在** 任何以 `AMZ_REPORT ` 开头的元素，则服务端沿用旧规则：

1. 取**最后一条非空**元素，去掉首尾空白；
2. 若该字符串在**忽略大小写**后等于 `success` → 任务**成功**；
3. 否则 → 任务**失败**，`failure_detail` 取该字符串（截断长度由服务端限制）。

**限制：** 旧规则无法上报 `environment`、无法由客户端指定精确的 `finished_at`（由服务端在结案时写入当前时间）。

## 4. 与服务端实现的对应关系

- 解析实现：`src/task_report_parse.py`（函数 `parse_task_report_footer`）。
- 结案与落库：`src/db.py` 中 `finalize_task_from_report`（更新 `status`、`failure_detail`、`finished_at`、`task_environment`）。
- 任务重试时，`task_environment` 会被清空，需在新一次上报中重新带上 `AMZ_REPORT`。
- **管理端联调**：登录管理员账号后，侧边栏 **「上报日志解析」**（路由 `/tools/task-report-parse`）可粘贴 `log_lines` 多行文本，调用 `POST /api/v1/admin/task-report/parse-preview`，返回与结案一致的解析字段：`success`、`environment`、`finished_at`、`failure_detail`、`used_amz_report`。

若本文与代码行为不一致，**以仓库内当前代码为准**，并应同步修订本文。
