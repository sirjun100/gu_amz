# 亚马逊运维系统 — 项目需求文档

## 1. 文档说明

| 项 | 说明 |
| --- | --- |
| 项目名称 | 亚马逊运维系统 |
| 文档用途 | 产品/功能需求说明，供设计与开发对齐 |
| 客户端背景 | EasyClick 客户端：启动上报设备 ID，心跳 `heart(device_id)`，约每 1 分钟一次 |
| 约束 | 本文档仅描述需求，不包含实现与代码 |

---

## 2. 目标与范围

在现有项目基础上演进为「亚马逊运维系统」：管理端负责设备、关键词、多类任务、美国地址与自动注册；客户端通过统一接口拉取任务、回传执行日志（含图片），服务端据此更新任务状态并支持失败重试策略。

**本期范围**：下列功能模块及客户端接口；不包含具体技术栈选型与部署细节（可在设计阶段补充）。

---

## 3. 功能需求

### 3.1 设备管理

**数据来源**

- 客户端启动或运行过程上报设备标识；心跳接口携带 `device_id`，周期约 1 分钟。

**管理端能力**

- **列表**：展示已上报设备；支持查询（筛选/搜索按产品约定）。
- **备注/别名**：可为设备设置便于识别的名称或备注（仅编辑展示信息，不以删除设备为主要操作）。
- **在线状态**：列表展示**最后在线时间**（由心跳或等价上报更新）。

**约束说明**

- 管理端侧重**查询与备注**；若需删除/禁用设备，可在详细设计中单独约定权限与数据保留策略。

### 3.1.1 任务中心（已实现方向）

- 独立 **任务中心** 页面：支持按 **设备**、**状态**、**任务类型** 筛选与分页列表。  
- **失败任务**：可查看 **失败详情**（摘要、完整日志、截图）；提供 **重试** 操作（仅失败状态），重试后任务回到待执行并清空该次日志与截图记录（磁盘文件一并删除）。  
- 管理端接口命名（示例）：`GET /api/v1/admin/task-center/tasks`、`GET /api/v1/admin/task-center/tasks/{id}`、`GET /api/v1/admin/task-center/screenshots/{image_id}`、`POST /api/v1/admin/task-center/tasks/{id}/retry`；设备下拉数据 `GET /api/v1/admin/devices/options`。

---

### 3.2 随机关键词管理

**能力**

- 批量导入：文本文件（`.txt`），**一行一个关键词**。
- 列表：**分页查询**。

**客户端接口**

- 提供 `get_random_keywords`（说明中原为 `get_randrom_keywords`，实现时建议统一拼写）  
  - 参数示例：`num=2`（一次返回的随机关键词条数，具体上限与默认值在设计中约定）。
  - 行为：从关键词池中随机选取指定数量返回（去重、池为空时的错误码等在设计中约定）。

---

### 3.3 搜索产品点击任务

**业务字段**

- 关键词  
- 产品标题  

**创建任务 — 界面与逻辑**

1. 输入关键词与产品标题。  
2. 展示可选设备列表（**多选**）。  
3. **任务量分配**（两种方式，可同时支持）：
   - **手动**：为每个已选设备指定要执行的任务数量。  
   - **智能**：输入**总任务数**（例如 100），系统按已选设备数量等方式**均分或按约定规则分配**到各设备（示例：10 台设备各 10 条）。  
4. 创建后进入任务列表，需能**查看执行进度**并**刷新状态**（轮询或 WebSocket 等由实现选定，需求层面要求「可跟进、可刷新」）。

---

### 3.4 相关产品点击任务

- 字段与交互**与「搜索产品点击任务」相同**（关键词、产品标题、设备多选、手动/智能分配、进度与状态刷新）。  
- 与 3.3 的差异仅为**任务类型**不同，便于客户端 `get_task` 按类型区分处理。

---

### 3.5 同类产品点击任务

- 同上，**与搜索产品点击任务界面与字段一致**，任务类型独立枚举。

---

### 3.6 自动注册任务

**业务字段**

- **手机号**：由管理端批量导入，**一行一个**。  
- **用户名**：从「地址管理」随机抽取一条记录，取该条 **姓名（recipient_name）**；若为空则自动生成占位用户名。  
- **密码**：服务端 **自动生成**（10 位：含大写、小写、至少 1 个特殊字符 `!@#$%^&*`、多位数字，再打乱顺序）。  
- **地址**：随机一条美国地址快照写入任务（`address_id` + `address_snapshot`）。

**批量新增**

- 支持 **智能均分** / **手动分配**：与点击类任务类似，按所选设备将 N 条手机号拆成各设备上的 N 条待执行注册任务；手动模式下各设备任务数之和须等于手机号行数。  
- 仅需导入 **手机号**（可选 Tab/逗号后附内容，仅取首列作为号码）。

---

### 3.7 地址管理

**数据**

- 初始数据来自仓库内 **`美国地址.xlsx`**，需**导入数据库**。  

**管理端**

- 支持 **Excel/文件导入**（与初始导入格式一致或可配置映射）。  
- 地址库 **增删改查**（CRUD）。  
- 为自动注册等模块提供随机选用能力（见 3.6）。

---

## 4. 客户端接口与任务流转

### 4.1 统一拉取任务 — `get_task`

- 上述各任务类型均通过**同一套拉取入口**（如 `get_task`）由客户端获取。  
- 客户端按**任务类型**及约定**顺序**请求/筛选（具体参数：设备 ID、类型、分页等在设计中定义）。  
- 客户端**成功领取某条任务后**，服务端将该任务状态更新为 **「正在执行」**（说明中的「重置/置状态」含义）。

### 4.2 执行结果上报 — 日志（含图片）

- 无论成功或失败，客户端均需**上报日志**。  
- 支持**附件：图片**（多张、大小限制、存储方式在设计中定义）。  

**成功判定（需求级规则）**

- 以日志内容为准：若**最后一条**日志表明结果为 **`success`**，则判定任务**成功**。  
- 否则由服务端结合规则判定为失败或其他终态（具体解析规则可在实现中细化）。

**服务端行为**

- 根据日志判定成功/失败后更新任务状态。  
- **失败**时，服务端**可配置重试**（次数、间隔、是否换设备等策略在设计/配置中约定）。

## 5. 非功能需求（建议项）

以下内容建议在详细设计阶段量化，本文档仅列方向：

- **可用性**：管理端操作反馈、列表分页性能。  
- **安全**：管理端鉴权、客户端设备身份校验、接口防滥用。  
- **一致性**：任务状态枚举与设备视角下的「未执行 / 成功 / 失败」分类一致。  
- **可观测**：关键操作与任务状态变更可审计（可选）。

## 6. 术语与待定事项

| 术语/事项 | 说明 |
| --- | --- |
| 任务类型 | 至少包含：搜索产品点击、相关产品点击、同类产品点击、自动注册（及是否还有「仅心跳」等非任务类型由实现确认） |

| 智能分配 | 除均分外，是否支持按设备负载、权重分配，本期若未约定则默认均分或按已选设备数整除分配 |
| 重试 | 失败重试次数、是否自动、是否人工介入，需在配置或二期需求中明确 |
